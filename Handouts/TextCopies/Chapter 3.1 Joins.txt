Using joins, you retrieve data from two or more tables 
based on logical relationships between the tables.
Joins indicate how data from one table is used to 
select the rows in another table.
The relationships may be explicitly defined 
in the database or may only be implicit as 
part of a query (ad hoc).

This is the result set. It is what is in common between the tables.
The result set consists of the rows from both tables that match

===== Page 7 =====
Esme Basnet CS486/586 Intro DBMS
Joins
Inner joins return rows only when there is at least one row 
from both tables that matches the join condition.
• Inner joins eliminate the rows that do not match with a 
row from the other table. 
Outer joins return all rows from at least one of the tables or 
views mentioned in the FROM clause, as long as those rows 
meet any WHERE or HAVING search conditions. 
• All rows are retrieved from the left table referenced with 
a left outer join, and all rows from the right table 
referenced in a right outer join. 
• All rows from both tables are returned in a full outer join.

Understanding the Three Types of Table Relationships
One-to-One Relationship
• A value in a table maps to 0 or 1 row in a table.
One-to-Many Relationship
• A value in a parent-table maps to 0 or more rows in a 
child-table.
Many-to-Many Relationship
• Many values in one table map to many values in another table.

The syntax for a union operation
    SELECT_statement_1
UNION [ALL]
    SELECT_statement_2
[UNION [ALL]
    SELECT_statement_3]...
[ORDER BY order_by_list]
UNION removes duplicate records (where all columns 
in the result set are the same), UNION ALL does not.
There is a performance hit when using UNION vs 
UNION ALL, since the database server must do 
additional work to remove the duplicate rows.
Rules for a union
• Each result set must return the same number of columns.
• The corresponding columns in each result set must have compatible 
data types.
• The column names in the final result set are taken from the first 
SELECT clause.
